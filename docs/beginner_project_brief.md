# Beginner MVP — 프로젝트 브리프

Claude Code 세션 시작 시 이 문서를 먼저 읽고 작업을 시작할 것.

## 서비스 개요
Forage 모델의 한국형 적응. 구직 대학생이 지원 전에 실제 업무 과제를 시뮬레이션으로
경험하고, 기업은 이를 통해 고관심 지원자를 발견하는 양면 플랫폼. 자기보고(성격검사)가
아니라 행동(실제 과제 수행)으로 직무 적합도를 드러낸다는 철학.

## 사이트 구조 — 반드시 3개로 분리, 절대 하나로 합치지 말 것

1. **구직자 사이트** (beginner.today)
   - 구직자가 가입 → 사전질문 → 직무 시뮬레이션 추천 3개 → 수행 → 제출
   - '지원하기' 누르면 groupby(외부 채용 사이트)로 이동. 지원해도 안 해도 기록은
     구직자 프로필에 남음.

2. **기업 사이트** (Beginner biz) — 구직자 사이트와 완전히 별개 사이트
   - 기업이 고유코드(unique_code)를 입력해 로그인
   - 자기 기업 시뮬레이션을 수행한 구직자들의 답안 + 프로필을 열람
   - 마음에 드는 지원자에게 이메일로 제안 가능
   - **주의**: 이 사이트는 구직자가 '지원'해서 연결되는 곳이 아님. groupby와도 무관.

3. **groupby** — 완전 외부 사이트. '지원하기' 버튼의 딥링크 대상일 뿐, 우리 DB와 무관.

## 확정된 핵심 결정사항

### 데이터 공개 정책: A안 확정
- 기업이 구직자의 답안 원문을 볼 수 있음 (Forage처럼 답안을 안 보여주는 방식이 아님)
- 단, **동의 기반**. 두 종류의 동의를 반드시 분리해서 구현:
  1. `discovery_consent` — 온보딩 마지막 단계. "기업이 나를 발견하고 이메일로
     제안을 보내도 되는지" 동의 (Forage 마지막 화면 "Are you ready to be
     discovered by top companies?" 대응)
  2. `answer_transmission_consent` — 시뮬레이션 제출 시. "이 답안을 [기업명]에게
     전송하는 것에 동의하시나요?" 동의. **이 값이 true인 제출물만** 기업 사이트에
     노출되어야 함 (company_visible_submissions 뷰 참고)

### 온보딩 / 사전질문: Forage 스타일 참고, 성향검사 아님
- "이건 테스트가 아니에요" 같은 온보딩 카피는 사용하지 않음 (이전 버전 서비스의
  컨셉이며 현재 서비스와 성격이 다름)
- 목적은 성향 파악이 아니라 "어떤 직무 시뮬레이션을 받고 싶은지"를 좁히는 것
- Forage 실제 사전질문 흐름을 참고 (첨부 스크린샷 기반):
  1. 학력 (Education) — 현재/최종 학력(단일), 전공(복수), 학점(optional)
  2. 관심 직무 (복수) — 서비스기획·PM / 개발 / 데이터 / 디자인 / 마케팅·그로스 / 운영·CS
     → 이게 추천 로직의 핵심 시그널
  3. 관심 기업 (복수, 로고 선택형) — 당근마켓, 직방, 야놀자, 무신사, 오늘의집, 컬리,
     카카오페이, NHN 등 실제 아웃리치 대상 기업 우선 노출
  4. 근무 선호 — 근무 가능 지역, 관심 고용형태(인턴/신입/계약), 이주 가능 여부
  5. Discovery 동의 화면 (위 discovery_consent)

### 추천 로직: 룰 기반만. IRT/Contextual Bandit 사용 안 함
- 이 MVP에서는 사용하지 않기로 확정 (기존 서비스에서 쓰던 인프라이지만 여기선 불필요)
- `관심 직무 × 관심 기업/도메인` 필터링으로 직무 시뮬레이션 3개 추천
- SQL 예시는 schema.sql 파일의 주석 참고

### 프로필(이력서): 최대한 심플하게. 아래 항목까지만
- **자동이력**: 완료한 시뮬레이션 목록 (직무명, 기업명, 완료일) — 자동 축적
- **한줄소개**: 텍스트 한 줄
- **외부링크**: GitHub, 포트폴리오, LinkedIn
- **프로필 사진**: 업로드/변경 (Supabase Storage `avatars` 버킷)
- **온보딩 답변 편집**: 학력/전공/학점/관심직무/관심기업/근무선호/이주가능여부/
  discovery_consent를 프로필 화면에서 직접 열람하고 수정 가능 (온보딩과 동일한
  UI 컴포넌트를 `src/lib/profile-fields.tsx`로 공유)
- **넣지 않는 것**: 11축 역량 신호 배지/레이더 차트 (이전 서비스 컨셉, 현재는 폐기),
  PDF 내보내기 — 전부 2차 이후로 미룸. 기능 과다로 인한 혼란 방지가 목적.

### 11축 채점 시스템의 위치
- MVP 라이브 플로우에는 자동 채점을 넣지 않음. 기업이 답안 원문을 직접 읽는
  구조(A안)이기 때문.
- 11축 채점 시스템(미션_평가기준.pdf)은 Junemo가 Claude에게 "평가 기준에 맞춰
  채점해줘"라고 요청할 때 쓰는 내부/개인 도구로만 유지. 서비스 기능 아님.
- DB에는 `submissions.score_json` 컬럼만 비워둠 (나중에 필요하면 확장 가능하도록)

## 데이터 모델
`beginner_schema.sql` 파일 참고 (이 브리프와 같은 위치에 있어야 함). 테이블:
`job_seekers`, `companies`, `job_simulations`, `submissions` + 뷰
`company_visible_submissions`.

## 다음 작업 순서 (제안)
1. Next.js 14 + Tailwind + Supabase 프로젝트 셋업, 위 스키마 마이그레이션
2. 구직자 사이트: 온보딩 5단계 → 프로필 → 추천 3개 → 시뮬레이션 수행/제출 화면
3. 기업 사이트: 고유코드 로그인 → company_visible_submissions 대시보드
4. 지원하기 버튼 → groupby 딥링크 (플레이스홀더 URL로 우선 연결)
5. (나중) 복붙 감지 · 제출시간 기반 검증 레이어 강화, genesislab 연동
